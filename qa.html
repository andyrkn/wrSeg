QA
Testing
Goal

Guaranteeing a level of quality for the end client, and to help the software development team to identify and signal problems/limitations early in the Development Process. How we do that ? We employ a combination of automated testing and manual testing.

How tests are done
Types of tests

There are different types of test that will be done, some automatic and some manual:

Usability tests (manual) -> checking the responsivness and the interface on different browsers, platforms etc.
Functionality tests (manual) -> making sure that each part of the application (buttons, links etc.) is working as intended
Correctness tests (automatic and manual) -> checking how accurate the results are and the response for invalid inputs
Performance tests (automatic) -> measuring how much time it takes to give the result

Technologies

For automatic tests we will use scripts written in Python 3.x and the following libraries:

  • Requests
  • Matplotlib
  • OpenCv

Requests library will be used for interacting with the website. We also thought of using urllib.request but we decided against it because Requests supports a fully restful API, has a built in json decoder and is overall easier to use because of its short and simple syntax.

Matplotlib will be used in automatic tests to create a visual representation of the application's performance and correctness. We choose it because is is easy to use and it is popular (therefore well documented).

OpenCV will be used to compare the visual result (eg. a portion of the image) with the expected output. We chose it because it is a high performance library and it is popular.

Input files

We will be using image files of different formats (JPEG, PNG, PDF, TIFF etc.) which will also be transformed (rotated, mirrored etc.). These files will be split in two groups: files used for automatic testing and files used for manual testing.

The files used for manual testing will be relatively complex and there will be quite a high chance that the application will give wrong results for them. These files usually contain handwritting, bad lightning etc.

The files used for automatic testing will initially be simple in order to test if the application works decently. After a while, some files used for manual testing (which are more complex) might be moved to automatic testing if the application performs well enough.

Implementation details
Segment comparer

Utility

This program is a component of the automated tests for the back-end algorithm. It computes a score for the similarty between two segments, this is useful in comparing the ouput segments, of the page segmentation algorithm, with target segments from a set of tests.

How it works

The output and target segment will be split in 3 other segments:

  • Common area segment (intersection between output and target)
  • Missing area segment (target minus output)
  • Extra area segment (output minus target)

For each of the above segments a set of tests will be run, each returning a score. After all the scores are computed a weighted sum of all of them will be returned, this final score in the [0, 1] interval represents a grade for each segment created by the segmentation algorithm. A threshold needs to be chosen on what is a acceptable output.

Known problems

The segment comparer returns false positives when the difference between the output and the target is another small segment, the page number or a small logo for example. The weights will have to be adjusted and more tests will have to be added to account for this and other extreme cases.
